- **Stated intent** (from design-doc diff):

## 2. Summary

Overall the change lands the right design. BUG-080 v2 is a well-justified reversal of the v1 position: the ordering constraints for the graceful reconcile point are spelled out in-code, `loadCdcCheckpoint(currentPrimary)` happens after `cdcCollector.stop()`'s final poll and before any mutation, drain warnings are preserved (no sensitivity dilution), and `catch (Throwable)` degrades cleanly to v1 drain-only semantics. BUG-081's `splitByDuplicateElementId` algorithm is O(n), order-preserving, and zero-overhead in the common case.

**One blocking issue**: unit test `largeBatchWithOneCollision_splitsIntoExactlyTwoSubBatches` asserts sub-batch sizes that the algorithm cannot produce — the test would fail at `mvn test`. Two cosmetic comments and one dead null-check are Minor.

### Major (should fix before merge)

- **M1** — `src/sync-applier/src/test/java/com/neo4j/ha/sync/applier/ChangeApplierSplitTest.java:148-166` — `largeBatchWithOneCollision_splitsIntoExactlyTwoSubBatches` asserts `out.get(0).size() == 51` and `out.get(1).size() == 50`, but the implemented algorithm only splits on the **second** occurrence of an element id. Trace of the test input `[u0..u49, eid-collision, u50..u97, eid-collision, tail]`:
  - The first `eid-collision` (index 50) is new to `seen` → no split.
  - The second `eid-collision` (index 99) triggers the only split.
  - Actual output sizes: **99 / 2** (not 51 / 50).

  The `assertEquals(2, out.size())` and the total-size invariant still pass (`51+50==101==99+2`), but the two size assertions will fail under `mvn test`. This is almost certainly why the commit was pushed through `-DskipTests` (matching the warning the BUG-080 design doc already records for this exact risk class at `docs/.../ha-agent-design.md` "mvn install -DskipTests" advice line).

  **Fix (pick one)**:
  - **(a)** Correct the assertions to reflect the algorithm: `assertEquals(99, out.get(0).size())` and `assertEquals(2, out.get(1).size())`, and adjust the comments to "50 uniques + first collision + 48 uniques" and "second collision + tail".
  - **(b)** If the intent really was "split on every `eid-collision` occurrence including the first", change the input to `[eid-collision, u0..u49, eid-collision, u50..u97, eid-collision, tail]` so the second occurrence is at index 51 and produces the documented 51/50 shape — but this changes the test's narrative ("one collision mid-way" → "two collisions") and weakens coverage.

  Strongly prefer (a): the existing algorithm is correct, only the assertion arithmetic is wrong.

### Minor

- **m1** — `src/ha-agent/src/main/java/com/neo4j/ha/agent/failover/FailoverOrchestrator.java:447-454` — the BUG-080 cursor-snapshot comment claims the snapshot must happen "BEFORE copyCdcCheckpoint (line below) rewrites the NEW primary's checkpoint key". `copyCdcCheckpoint(currentPrimary, newPrimary)` reads the OLD key and writes the NEW key (`CheckpointManager.java:181-193`); the snapshot also reads the OLD key (`loadCdcCheckpoint(currentPrimary)`), so the two cannot race on the same key. The real invariant is "after `cdcCollector.stop()`'s final poll, before any re-start of CDC" — which the current placement satisfies for a different reason. **Fix**: reword to "after `cdcCollector.stop()`'s final-flush poll (so `lastRelTs` / `lastDeleteTs` reflect the drain horizon), before any code path that could re-start CDC or clear the OLD key". The "either node's checkpoint can't race us" defense-in-depth line further down is fine.

- **m2** — `src/sync-applier/src/main/java/com/neo4j/ha/sync/applier/ChangeApplier.java:75-81` — the new split metric increment is guarded by `metrics != null`, but pre-existing `metrics.syncEventsApplied.increment(events.size())` on line 90 (post-loop) and `metrics.syncApplyErrors.increment()` on line 95 have no such guard, and the constructor (`ChangeApplier.java:28`) requires `metrics` as a non-null injected argument. Either `metrics` is never null (remove the guard — dead code) or it *can* be null under a test harness (in which case the other two increments NPE). **Fix**: drop the `metrics != null` guard to match the surrounding contract; if a future test harness needs to pass null, add the guard consistently to all three increments.

- **m3** — `src/sync-applier/src/main/java/com/neo4j/ha/sync/consumer/PendingRecovery.java:78-82, 152-174` — the `BUG-081 DIAG` batch-dump log is correctly scoped (≤10 lines per PEL recovery, `log.isInfoEnabled()` gated, explicit TTL comment "Remove once BUG-081 root cause is confirmed"). **Fix**: once BUG-081 plan A is validated by chaos runs that show `batchSplitForDuplicateElementIdTotal > 0 && rel_miss == 0`, open a cleanup ticket to remove `logBatchForBug081` and its call site. Nothing to change now; flagged to avoid the log living forever.

- **m4** — `scripts/deploy/ha-load-switchover-test.py:757-835` — `wait_rel_count_converged` opens and closes a Neo4j driver per poll per node (3 drivers/sec × up to `repl_sync_timeout` seconds). Functionally fine but noisy (fresh bolt handshake each tick) and inconsistent with the rest of the script's pooled-driver pattern. **Fix**: pre-create one driver per node outside the loop, reuse them for `.single()` polls, close at the end — same algorithm, ~1 ms per poll instead of ~10-30 ms.

## 5. Strengths (值得表扬)

- `FailoverOrchestrator.java:502-536` — the ordering-constraint comment block is the gold standard for this codebase: it enumerates four independent invariants (switchTarget live, cleanup-must-wait, applier-stopped, writes-blocked) and the rationale for each, so any future reordering must explicitly address all four.
- `ChangeApplier.splitByDuplicateElementId` (`ChangeApplier.java:125-143`) — O(n), single-pass, order-preserving, zero overhead on the no-collision path, and the docstring explicitly calls out the "per-event splitting would pay extra round-trips" and "group-by-label would leave the hazard intact" alternatives that were rejected. That makes the choice defensible against future drive-by refactors.
- `wait_rel_count_converged` docstring (`ha-load-switchover-test.py:763-787`) — clearly explains why rel-count cannot use an absolute target the way node-count can, and why a single cross-node agreement isn't enough (temporal stability check). This pre-empts a class of false positives that would otherwise look identical to real BUG-080 / BUG-081 regressions in the report.
- Design-doc BUG-081 root-cause analysis (`ha-agent-design.md` BUG-081 section) — the stream-id / xpending / cypher-shell evidence chain is reproducible; the three-outcome decision table for the DIAG log is actionable. This is how incident writeups should look.
- BUG-080 v2 preserves v1's warning signals intact (`Phase 2.6 afterAsync drain did NOT confirm` still fires) while adding `replayed>0` as a quantitative drift signal — exactly the "both layers observable" pattern the v1 lesson warned against losing.
